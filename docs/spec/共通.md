## 2. 認証・権限

### 2.1 ロール定義

| ロール | 説明 |
|--------------|-----------------------------------|
| システム管理者 | システム全体の管理・全データへの操作権限 |
| 管理         | 組織全体の管理・全データへの操作権限     |
| 営業         | 担当案件・エンジニアデータの閲覧・一部操作 |
| エンジニア    | 自分に関連するデータの閲覧・操作     |

### 2.2 権限チェック
- 権限判定はUIとAPIの双方で実施する
- 操作不可はUI上で非活性（disabled）または非表示とする
- 権限のないAPIリクエストは `403 Forbidden` を返却する

## 4. バリデーション方針

- フロント（zod）とバック（Laravel）の双方でバリデーションを実施する
- エラーは各入力項目の直下に表示する
- 保存操作はバリデーション通過後のみ実行可能とする

### 4.1 共通バリデーションメッセージ例

| ケース | メッセージ |
|---|---|
| 必須項目未入力 | `[項目名]を入力してください` |
| 最大文字数超過 | `[項目名]は[N]文字以内で入力してください` |
| 形式不正 | `[項目名]の形式が正しくありません` |

---

## 5. データ管理方針

### 5.1 削除方式
- 原則として**論理削除**を採用する
- 物理削除が必要な場合は各機能仕様書に明記する

### 5.2 楽観的ロック
- 同一データへの競合更新は楽観的ロックで制御する
- 競合検知時は `409 Conflict` を返却し、UIでは確認ダイアログを表示する
    - メッセージ: 「別のユーザーが更新を行いました。更新内容を確認し編集を続けますか？」

### 5.3 監査ログ
以下の操作を記録する。

| 記録項目 | 内容 |
|---|---|
| 操作日時 | JST基準 |
| ユーザーID | 操作実行ユーザー |
| 操作種別 | CREATE / UPDATE / DELETE |
| 対象リソース | テーブル名 + ID |
| 変更内容 | 変更前後の値 |

## 6. パフォーマンス要件（UX SLO）

| 処理 | 目標 | 上限 |
|---|---|---|
| 一覧初期表示 | 1秒 | 2秒（3秒超でローディング継続表示） |
| フィルター・ソート | 1秒 | 2秒 |
| 登録・更新 | Optimistic UI（即時反映） | 失敗時はロールバック＋エラー表示 |
| 重い処理（10秒超見込み） | 非同期化 | 完了通知を表示 |

---

## 7. エラー表示・通知方針

| ケース | 表示方法 |
|---|---|
| バリデーションエラー | 入力項目直下にインライン表示 |
| 操作の成功 | 画面下部にトースト表示 |
| API通信エラー・重大エラー（5xx等）・通信エラーメッセージ | エラー画面遷移（エラーメッセージ + ホーム画面へのリンク配置） |

---

## 8. アクセシビリティ要件

| 項目 | 基準 |
|---|---|
| 色のコントラスト比 | WCAG 2.1 Level AA（通常テキスト 4.5:1 以上） |
| 色覚対応 | 色のみに依存しない情報伝達（アイコン・テキスト併用） |
| キーボード操作 | 全機能をキーボードのみで操作可能 |
| 対応範囲 | デスクトップ（PCブラウザ）のみ / スマートフォン・タブレット対応は対象外 |
